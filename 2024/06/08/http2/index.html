HTTP 2.0 简单介绍

2024-08-22 · 2 min · 667 words · 基础知识

HTTP 1.1 - 1 次 1 份

1 次 1 份：必须等待请求处理完成后，才能继续发送下一个请求。

类似于窗口排队办理业务，如果前面的人业务没有办理完成，后面的人就无法办理业务。

问题： 职能串行处理请求，如果某个请求耗时很久，则会阻塞后续的请求。

解决办法：

  1. 增加并发请求的数量：chrome 默认并发请求数量是 6，可以通过 chrome://net-internals/#quic 配置。

  2. 减少请求的数量：雪碧图、内联 JS 、内联 CSS 等。

  3. 使用多个 CDN 域名：浏览器的连接数量限制是针对一个域名，所以将资源分发到不同的 CDN 域名，可以变相提高并发请求的数量。

HTTP2 多路复用

HTTP2 的多路复⽤原理是指在同⼀个 TCP 连接上，可以同时发送和接收多个请求和响应，⽽不需要按照顺序⼀个接⼀个地处理。

类似于窗口叫号处理业务，先处理完的业务可提前叫号。

实现原理：

  1. 重新设定了 HTTP 的传输格式，由之前的报文格式改成帧数据

  2. 将传输内容分段成⼀个⼀个帧数据，每个帧数据都有⼀个流标识符，⽤于标记帧数据哪个流

  3. 客⼾端和服务器接可以不按顺序发送和接收这些流信息，最终将流信息拼接成完整的内容

HTTP 1.1 和 HTTP 2 请求⽐较

HTTP2 服务器推送

因为每个请求都有标识区分，所以服务器可以实现主动向客⼾端推送资源，⽐如当请求 index.html 时，把 index.html 内引⽤的资源⽂件⼀并推送给客⼾端。

弊端：

  1. ⽤⼾可能只是点错了，但是服务器就主动发送了很多资源给客⼾端，⽆形中增加了客⼾端缓存。

  2. 客⼾端和服务器请求数量不对等，容易受到 DDoS 攻击。

HTTP2 头部压缩

HTTP 1.1 每次请求都会携带请求头数据，并且头数据还不⼩，每次请求和响应中这些数据中存在很多重复的内容，尤其是 Cookie 和 User-Agent 这些不怎么改变的内容，却⼀直占⽤着请求资源。

HTTP2 采用 HAPCK 算法进行压缩

简单理解就是，客户端和服务端都维护一个字典，字典中存储了请求头中重复的部分，当请求头中包含重复的部分时，只需要发送字典的索引，服务端接收到字典的索引时，直接从字典中获取，减少请求头数据传输。
